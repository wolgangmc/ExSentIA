9.7   Sorting and bundling output

    The sorting and bundling process allows you to sort customer data and to place multiple customer communications to the same address into one envelope (also called householding).

    After sorting the data stream within Orchestrator or with a third-party program, you can then complete the process in Orchestrator.

    Because much of the sorting and bundling process is dependent on the timing of when resources are used, you must apply optimization settings to identify how and when your communication and resources are used for sorting and bundling process. You can use synchronous or asynchronous Orchestrator job flows to sort the data, or you can manually sort data with a third-party program of your choice. For asynchronous and custom flows, you must run the final post-sort job in orchestration to generate the final sorted and bundled output. Synchronous job flows perform pre and post-sort processing within the same job.

    The following section provides a basic overview of the sorting and bundling process:

    The sorting and bundling process includes the following steps:

    1. Set up a sort index data source – In your communication, you define a sort index data source that uses a sample file with a columnar layout.

      For more information about defining a sort index data source, see Section 9.7.1 “Setting up a sort index data source”.

    2. Pre-sort processing – After you have set up your sort index data source, you must set up your communication for production using pre-sort processing (also called pre-sort, presort, or Step One). During pre-sort processing, the engine produces customer data that can then be manipulated to sort and update the files. For more information about pre-sort processing, see Section 9.7.2 “Setting up a communication for pre-sort processing”.

      The communication can also be configured to perform bundling, which is a convenient way to group customer documents. You can group customer documents in a number of ways, such as a group of customers at one address (also known as householding) or in a hierarchical arrangement. When you are producing recipient copies, recipient document sets are indexed separately from customer document sets so that you can sort and bundle recipient document sets appropriately based on the recipient data. For more information about bundling, see To use bundling to group customer information.

      For more information about recipient copies, see section 4.11 "Setting up recipient profiles" in OpenText Communications - Web Client User Help (CCWC250100-H-UGD)section 4.11 "Setting up recipient profiles" in OpenText Communications - Web Client User Help (CCWC250100-H-UGD).

    3. Sorting – Using the files that were generated from the pre-sort run or runs you reorder customer information for the best postal discounts. You can use synchronous or asynchronous Orchestrator job flows to sort the data, or you can manually sort data with a third-party program of your choice. You can sort output based on information from customer driver files in many ways (for example, by state and ZIP Code).

      For more information about sorting, see Section 9.7.4 “Sorting data files”.

    4. Post-sort processing – After the sorting process, you run the post-sort processing (also called post-sort, postsort, or Step Two). The engine uses the data in the sort data file(s) to produce final output according to the changes in the sorted index file and other communication settings. The engine then produces output based on the sorted information.

      For more information about post-sort processing, see Section 9.7.5 “Running engine orchestration for post-sort processing”.

    Note

    Duplex settings are not supported with sorting and bundling. If communications or documents that are included in the sorting and bundling process include duplex settings, these settings will be ignored.
