    To provide an example of a typical tenant configuration, the sample tenant contains the following domains:

    • Testdomain—Defined as a non-production domain. Batch engine runs do not increment the transaction count for the temporary product key. Output generated from all engine runs in this domain will contain the demo watermark.

    • Proddomain—Defined as a production domain. Batch engine runs in this domain will increment the transaction count for the temporary product key. Output generated from all engine runs in this domain will contain the demo watermark.

    Important

    In an active production domain, the output generated from an engine run does not contain the demo watermark. However, the production domain in the sample tenant does not support this option.

    To enable user access, the sample tenant also includes default OTDS users and groups with specific permissions for various features of the OpenText Communications deployment.

    Default user permissions for OpenText Communications web client and DAS

    In each domain in a tenant, OTDS user group and OpenText Communications role associations determine the level of access each user has when they sign in to DAS from the OpenText Communications web client.

    The sample tenant includes default OTDS  user groups that are used for user authentication, and also has a set of default OpenText Communications roles with defined access permissions for that tenant. The level of access in DAS for a user is determined by the OpenText Communications role that is assigned to the OTDS group for that user. For users in multiple OTDS groups, the level of access is determined by the group with the highest permissions.

    Not all OTDS groups have an assigned OpenText Communications role by default. If an OTDS group does not have an assigned role, then users who are members of only that group cannot sign in to DAS from the OpenText Communications web client.

    Default user permissions for Interactive

    OTDS user groups also determine the level of access each user has when they sign in to the Interactive Console or Interactive. For a user in multiple OTDS groups, the level of access is an accumulation of all groups to which the users belongs.
