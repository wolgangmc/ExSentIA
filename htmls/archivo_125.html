    When a resource moves through the approval workflow, it is assigned incremental whole numbers for versioning. Each time a new draft is created from an approved resource, the version number for that resource is incremented. For example, when a designer creates a new communication resource, this is version 1 of the resource, and this version is in draft state. When a reviewer approves this resource, version 1 of the resource is in the approved state. If a designer then creates a new draft of the approved resource, then that draft is version 2 of the resource. However, if a reviewer rejects version 1 of the resource, and a designer then returns it to a draft state, the version number of the resource is not incremented.

    Viewing a specific version of a resource

    By default, when you select a resource in the asset library panel, the Properties panel displays information about the latest version of that resource. If a resource is in draft state and has multiple versions, a version selector next to the resource name in the Properties panel displays the version number of the selected resource.

    To see a specific version of that resource, click the version selector, then select the version that you want to see.

    Promoting a resource

    If you want to revert to an older version of a resource, you can select that version and promote it in order to make it the latest version.

    Note

    You cannot promote a resource if the latest version of the resource or the version that you want to promote is locked by another user.

    When you promote a resource, keep the following considerations in mind:

    • If the latest version of the resource is in the approved state, a new draft version of the resource is created and has the same content as the older version that you promote. The previous version is still available in the asset library.

    • If the latest version of the resource is in the draft state, the content of the latest version is replaced with the older version that you promote but the version number of the resource is not incremented.

    To promote a selected resource:

    1. In the Properties panel, click the version selector, then select the version that you want to promote.

    2. Click the Promote button, then click Yes to confirm your action.

      The selected version is promoted and becomes the latest version.

    Managing application packages and subpackages

    Keep the following rules in mind when you create, update, delete, or upload application packages and their subpackages:

    • An application package can be associated with multiple subpackages, and a subpackage can be associated with multiple application packages.

    • Different versions of an application package can be associated with different sets of subpackages.

    • Different versions of a subpackage are always associated with the same application package.

    • Subpackage changes do not increment the version number or change the workflow state of their associated application package. For example, you can create, update, delete, or upload a subpackage without changing the version number or workflow state of its associated application package.

    • You can delete an application package only after you delete its subpackages. For information about deleting other resources, see Deleting resources.

    • Associating an application package with a domain does not automatically associate its subpackages with that domain. For example, an application package can have five subpackages in one domain and three subpackages in another domain. For other exceptions that apply when associating application packages or subpackages with domains, see Associating resources with domains.

    Tip

    To see package and subpackage associations:

    1. In the Orchestration asset library view, in the Packages panel, select a subpackage and then, in the Properties panel, select the References tab.

    2. Select Uses. You see the one associated application package listed.

    3. Repeat step 1 for the associated application package.

    4. Select Used by. You see one or more subpackages.

    To add a subpackage to an application package:

    1. In the Orchestration asset library view, in the Packages panel, select an application package.

    2. In the Properties panel, in the Linked subpackages list, click the Add button.

    3. In the Package dialog box, select the check box for each subpackage that you want to link to the application package, and then click OK.

    4. Click Save.

    Tip

    If an application was packaged in Design Manager with Enable split communication option in Content Author selected, and if subpackages for Single Document targets were created from the application, then, in DAS, you can link the subpackages to the application package to generate a Content Author communication container template and associated Content Author document templates.

    For more information, see Section 8.2.1.1 “Generating templates for Content Author communication containers and documents”.

    For more information about creating Content Author communication containers and documents, see section 1.1.2.1 "Creating a communication container with documents" in OpenText Communications - Content Author User Help (CCCA250100-H-UGD)section 1.1.2.1 "Creating a communication container with documents" in OpenText Communications - Content Author User Help (CCCA250100-H-UGD).

    To remove subpackages from an application package:

    1. In the Orchestration asset library view, in the Packages panel, select an application package.

    2. In the Properties panel, in the Linked subpackages list, select the subpackages that you want to unlink from the application package.

    3. Click the Remove button.

    4. Click Save.
