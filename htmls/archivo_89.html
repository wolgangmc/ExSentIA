    When the Orchestration service executes a flow model that contains tracking-enabled communications or output channels, event data from OpenText Communications is sent to OpenText Core Journey. This event information is sent through a JSON file, with the following attributes:

    • The schemaType parameter is set dynamically depending on the event type. The following table lists the supported event types. The examples that follow provide the sample JSON for an email event and a file channel event.

      Event type Schema type Event description

      Document Generation event

      OpenText Communications-Generated

      Document generated.

      Delivery events

      OpenText Communications-Email-Submit

      Email delivered through a Notifications email channel.

      The email is submitted to the OpenText Notifications Service for processing.

      OpenText Communications-<channelName>

      Document delivered through any output channel, except the Notifications email channel.

      The communication is delivered to its intended destination. For example, for an output file delivered using the File channel, this indicates that the file has been created in the specified output directory.

      Delivery failure events

      OpenText Communications-<channelName>-Failed

      Document delivery through an output channel failed.

      OpenText Notifications events

      OpenText Communications-Email-POST

      Email posted to recipient.

      OpenText Communications-Email-OPEN

      Email opened by recipient.

      OpenText Communications-Email-<status>

      Email delivery status. For example, OpenText Communications-Email-NDLV indicates that the email was not delivered.

    • The schemaSource parameter is set based on the schema source name that is provided in the OpenText Core Journey configuration for the corresponding domain.

    • The user namespace includes information from the output manifest, which is a structured data file that contains a comprehensive list of the output composed in an engine run. By default, the file describes the relationship between customers that are included in an engine run and the output files that are generated during that run. The output manifest can also include additional customer-specific data—such as customer name, email address and so on—or additional information about the communications that are being processed, such as campaign IDs. This data can come from the following sources:

      • Information on the Delivery tab of any object properties in Design Manager

      • Information that is defined on the Delivery tag of email designs in Designer

      • Metadata that is defined on output queue objects in DAS or in Design Manager

      • Metadata that is defined on DAS resources, and that is marked for inclusion in the output manifest.

      For information about defining metadata in Design Manager, see section 1.22 "Using metadata to tag objects in Design Manager and Designer" in OpenText Exstream Design and Production - Design and Import Help (CCDPDI250100-H-UGD)section 1.22 "Using metadata to tag objects in Design Manager and Designer" in OpenText Exstream Design and Production - Design and Import Help (CCDPDI250100-H-UGD). For more information about defining metadata in DAS, see Applying metadata tags to resources.
