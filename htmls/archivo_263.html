    After you publish your flow model and flow context, you are ready to initiate the production engine run as defined in the published flow resources. Depending on the type of input channel that you are using, and the type of communication being fulfilled, you can initiate the engine run in the following ways:

    Using a directory input channel

    When you use a directory input channel, you simply place your input file in the specified input directory that is defined in the flow context. This initiates new orchestration jobs with flow instances of any published flow models that use the input channel. Keep in mind, that this triggers orchestration jobs only if the input file name matches the file name pattern that is defined for that directory channel.

    For more information about defining directory input channels, see section 2.2.5 "Directory channel" in OpenText Communications - Orchestrator User Help (CCMCEORC-H-UGD)section 2.2.5 "Directory channel" in OpenText Communications - Orchestrator User Help (CCMCEORC-H-UGD).

    Sending data to an HTTP(S) input channel

    You can use an HTTP post request to trigger the input service that is associated with an HTTP(S) channel.

    The following request URI is used to post data to an HTTP(S) channel:

    orchestrationURL/api/v1/inputs/mode/domainID/serviceName

    orchestrationURL

    Base URL to the multi-tenant HTTP channel for orchestration requests. For example:

    http://localhost/orchestration

    mode

    Processing mode (batch or ondemand).

    domainID

    ID of the domain to access.

    serviceName

    Name of the service to access.

    The application that posts data must be signed in to the same tenant as the HTTP(S) channel. Like any other API call, an OAUTH2 authentication token must be provided either as a header or in a cookie. To get this token you can, for example, make the following type of post and then get the access token from the response:

    http://<otdsHostName>/otds/otdsws/otdstenant/<tenantName>/oauth2/token?grant_type=client_credentials&scope=otds:groups otds:roles

    For cross-site post requests, the header X-Xsrf-Token must also be provided. To get the value for this header you can, for example, make an orchestration/version API call and then get the XSRF-TOKEN value from the response cookie.

    When a request is made, the data in the HTTP body is sent as input to the HTTP(S) channel specified in the request URI, and the headers defined in the request will be available as variables in the flow model.

    For more information about sending data to HTTP(S) input channels, see section 2.2.9 "HTTP(S) channel" in OpenText Communications - Orchestrator User Help (CCMCEORC-H-UGD)section 2.2.9 "HTTP(S) channel" in OpenText Communications - Orchestrator User Help (CCMCEORC-H-UGD).
