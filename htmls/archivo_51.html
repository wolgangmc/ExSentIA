2.3.1   Configuring roles and permissions

    The OpenText Communications deployment uses OTDS for user authentication. To sign in to the OpenText Communications web client, a user must be a member of an OTDS group that is assigned an OpenText Communications role with the required permissions.

    Roles are always tenant-specific; however, access permissions are defined separately for each domain in a tenant. For each domain in a tenant, you configure user permissions by adding OTDS groups and assigning roles to those groups. When a user signs in to the OpenText Communications web client, their level of access is determined by the roles that are assigned to the OTDS groups to which they belong. For a user in multiple OTDS groups, the level of access is defined by the superset of permissions for all of the groups to which the users belongs.

    Depending on which features (design tools) are associated with the product key, the process of allocating users to a product key also grants them permission to utilize one of the Named User Seats (known as reserved seats in OTDS) for the web client design applications (Designer, Content Author, and Interactive). However, it does not automatically secure the seat until the user accesses a feature for the first time. Once a user is allocated to a seat, that seat stays reserved to that user unless revoked by an administrator in OTDS.

    Note

    Deleting a user does not automatically release the seat that is assigned to that user. A seat must be revoked in order for the seat to be available for reallocation.

For information about OTDS users, groups, and partitions, or how to allocate or revoke reserved seats, see OpenText Directory Services - Installation and Administration Guide (OTDS-IWC)OpenText Directory Services - Installation and Administration Guide (OTDS-IWC).
