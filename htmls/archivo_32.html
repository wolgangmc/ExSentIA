    When you first install your OpenText Communications deployment, the Helm chart includes the default configuration that creates the system tenant in the deployment. If you choose to automatically manage schemas in your OpenText Communications databases, the default configuration also creates a sample tenant environment.

    After you initially deploy OpenText Communications, you can use the following methods to create additional tenants in your deployment:

    • Tenant creation wizard—You use System management view in the OpenText Communications web client to access the tenant creation wizard. The wizard provides a convenient user interface to set up each tenant in the OpenText Communications deployment, and ensures that the tenant schemas are associated correctly with each service in the deployment. If you use the tenant creation wizard, you individually create each tenant in OpenText Communications, and then use Tenant management view to configure each tenant environment.

    • Bootstrap configuration—You use a bootstrap configuration with a Helm command to create and configure new tenants. The bootstrap configuration is a predefined parameter block that contains custom tenant configuration parameters. You can specify the bootstrap configuration parameters in the values.yaml file that you used to initially deploy OpenText Communications or you can create a separate bootstrap YAML file. Using a bootstrap configuration lets you automate the tenant setup process with a helm upgrade command. When you specify your bootstrap configuration parameters, you can also customize and select specific parts of the tenant configuration process that you want to automate.

    Note

    • You can use the OTDS REST API to create inherited OTDS tenants, which is the type of OTDS tenant that is used with OpenText Communications.

      To access the OTDS tenant REST API, use the following endpoint:

      {OTDS URL}/otdsws/api/index.html?tenants

    • You can also use a custom bootstrap configuration with the helm install command to create and configure individual tenants during the initial deployment. For more information about using bootstrap files during deployment, see OpenText Communications - Cloud Deployment Guide (CCMCE250100-IGD)OpenText Communications - Cloud Deployment Guide (CCMCE250100-IGD).

    Regardless of which tenant creation method you use, the manner in which you set up a tenant environment is determined by the schema management method that you selected when you deployed OpenText Communications. The following table lists the steps that are typically required to create tenants and set up a tenant environment using each method.

    Table 2.1: Tenant creation options

    Schema management Tenant creation wizard Bootstrap configuration
    Automatic schema management

    1. In the tenant creation wizard, create the tenant for OTDS and for each of the OpenText Communications services.

      On the wizard page for each OpenText Communications service, do one of the following:

      • Let the wizard automatically create the schema name and database credentials for the service.

      • Enter the existing schema name and database credentials for the service.

      In either case, the wizard associates the resource tables in the schema with the tenant.

    2. Install a product key in the OTDS tenant to make the tenant environment available for use.

    3. In Tenant management view, configure roles and domains to allow users to work in the tenant.

    1. Create a custom bootstrap configuration that includes the tenant information for each of the OpenText Communications services.

      For each OpenText Communications service, do one of the following:

      • Let the Helm command automatically create the schema name and database credentials for the service.

      • In the bootstrap configuration YAML file, enter the existing schema name and database credentials for the service.

      In either case, the Helm command associates the resource tables in the schema with the tenant.

    2. Run a helm upgrade command with your bootstrap configuration parameters.

      If you omitted the schema name and credentials for a service, the helm upgrade creates a new schema with credentials.

    3. Install a product key in the OTDS tenant to make the tenant environment available for use.

    Manual schema management

    1. Set up empty tenant schemas in the database for each of the OpenText Communications services.

    2. Assign a schema owner for each schema. At a minimum, the schema owner must have CRUD permissions (Create, Read, Update, and Delete).

    3. Run Liquibase to create tables in each of the schemas in your database.

    4. In the tenant creation wizard, create the corresponding OTDS tenant, and then associate the tenants for each service.

    5. Install a product key in the OTDS tenant to make the tenant environment available for use.

    6. In Tenant management view, configure roles and domains to allow users to work in the tenant.

    1. Set up empty tenant schemas in the database for each of the OpenText Communications services.

    2. Assign a schema owner for each schema. At a minimum, the schema owner must have CRUD permissions (Create, Read, Update, and Delete).

    3. Run Liquibase to create tables in each of the schemas in your database.

    4. Create a custom bootstrap configuration that includes the tenant schema information for each of the OpenText Communications services.

    5. Run a helm upgrade command with your bootstrap configuration.

    6. Install a product key in the OTDS tenant to make the tenant environment available for use.
