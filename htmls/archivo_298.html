    After you perform all of the steps necessary to initially deploy OpenText Communications, you'll need to configure the deployment for your organization’s needs.

    All OpenText Communications deployments use tenants and domains to keep data separate between business groups. Tenants contain data-isolated schemas for the OpenText Communications database services in your deployment, such as DAS, Orchestration, and Interactive. Tenants also contain domains that divide users and content into logical groups such as testing and production.

    An OpenText Communications tenant has the following properties:

    • A tenant name.

    • An associated OTDS tenant with the same name. The OTDS tenant manages the users, groups, and licenses for the OpenText Communications tenant.

    • Schemas for OpenText Communications database services. Each schema in a tenant holds the resource tables for a database service. An OpenText Communications tenant requires at least one schema so users can access the tables that the schema contains. A database service can have schemas in multiple tenants, but the schema names and database user names must be different for each tenant.

    To set up an OpenText Communications tenant, you will need to perform these steps:

    1. For each service to associate with the tenant, create a schema name, database user name, and database password in the service database.

    2. In each schema, create database tables for the service resources.

    3. Associate each schema and its tables with the tenant that you are configuring.

    You can manually create the schemas and tables in steps 1–2 in the databases. You either edit the databases directly or run a Helm command with a tenant bootstrap configuration. Then, you sign in to the OpenText Communications system tenant and use the tenant creation wizard to perform the associations in step 3.

    Note

    If you are using manual schema management with a Postgres database, you also need to grant the schema owner access to the system schema user by running the following command:

    GRANT <new schema user> TO <current system schema user>;

    However, if your deployment has automatic schema management enabled, you can use the following semi-automatic or automatic method to set up an OpenText Communications tenant:

    • Semi-automatic method: Perform only step 1 in the databases. You can either edit the databases directly or run a Helm command with a tenant bootstrap configuration. Then, in the tenant wizard, specify the existing schema names and database credentials and let the wizard perform steps 2–3.

    • Automatic method (default): Let the tenant wizard perform steps 1–3 for you.

    Note

    Automatic schema management is enabled if the autoManage.schema.enabled parameter was set to true in the configuration YAML file when OpenText Communications was initially installed or upgraded. For more information about the autoManage.schema.enabled parameter, see OpenText Communications - Cloud Deployment Guide (CCMCE250100-IGD)OpenText Communications - Cloud Deployment Guide (CCMCE250100-IGD).

    Important

    To help ensure security, enable automatic schema management and let OpenText Communications create the tables automatically, even if the tables are created from existing schemas. Before OpenText Communications creates the tables, it verifies that the existing schema name and database user name are unique across tenants as required.

    For more information about creating tenants, see Section 2.2 “Setting up tenants in OpenText Communications”.

    Regardless of which tenant configuration method you select, the system tenant is configured automatically when OpenText Communications is first deployed, and is used to manage other tenants and engines in the OpenText Communications deployment. In the system tenant, you use the System management view to create and manage tenants, to upload the OpenText Communications engines that are used in all other tenants, and to configure global scheduled tasks. This tenant forms a purely administrative environment; it cannot be used for creating and testing communications. To access the system tenant, you must be a member of the otadmins group in the OTDS system tenant.

    After you create tenants, you can access the Tenant management view for a tenant and then configure its domains, user roles, application properties, and categories. To manage a tenant, you must be a member of the tenantadmins group in the corresponding OTDS tenant.

    The following table describes the steps to access each view.

    View Steps

    System management view

    Sign in to the OpenText Communications web client as an OTDS administrator user at the following URL:

    https://<host>:<optionalPort>/design?tenant=system

    Tenant management view

    1. In the OpenText Communications web client, sign into the following URL as a tenant administrator user:

      https://<host>:<optionalPort>/design?tenant=<tenantName>

    2. Click the Tenant management button .
