2.3.2.5   Scheduling tasks for a domain

    1. In the Domains panel in Tenant management view, select the domain for which you want to add a scheduled task.

    2. In the Scheduled tasks tab in the Properties panel, click the Add scheduled task button .

    3. In the Scheduled task dialog box, specify a name for your task, then specify a cron expression and a type of action:

      Expire sorting assets

      This task expires the sort data and sort index data sources from a pre-sort run and those stored in the sorting repository. This is used to prevent outdated sorting assets from being processed. Once expired, the assets are not processed anymore.

      When you select this action, the following additional options become available:

      • Created—Specify the creation dates for the orchestration jobs that you want to process.

        • More than ... days before—Specify the number of days that an orchestration job was created before the scheduled task.

        • More than ... hours before—Specify the number of hours that an orchestration job was created before the scheduled task.

        • More than ... minutes before—Specify the number of minutes that an orchestration job was created before the scheduled task.

        • Fixed dates—Select a specific range of dates during which an orchestration job was created. When you select this option, you can use the calendar controls in the From and To fields to select specific dates.

      • Flow model—Specify the ID of the flow model in the pre-sort run that produced the sorting assets that should be expired.

      • Communication—Specify the ID of the communication in the pre-sort run that produced the sorting assets that should be expired.

      • Pre-sort queue name—Specify the name of the pre-sort queue in the pre-sort run that produced the sorting assets that should be expired.

      Mark jobs for deletion

      When you select Mark jobs for deletion, any qualifying jobs are tagged for removal but not removed from the database. When you select this action, the following additional options become available:

      • Start—The Start options let you specify the start dates for the orchestration jobs that you want to process.

        • More than ... days before—Specify the number of days that an orchestration job started before the scheduled task.

        • More than ... hours before—Specify the number of hours that an orchestration job started before the scheduled task.

        • More than ... minutes before—Specify the number of minutes that an orchestration job started before the scheduled task.

        • Fixed dates—Select a specific range of dates during which an orchestration job started. When you select this option, you can use the calendar controls in the From and To fields to select specific dates.

      • Finish—The Finish options let you specify the completion dates for the orchestration jobs that you want to process.

        • More than ... days before—Specify the number of days that an orchestration job completed before the scheduled task.

        • More than ... hours before—Specify the number of hours that an orchestration job completed before the scheduled task.

        • More than ... minutes before—Specify the number of minutes that an orchestration job completed before the scheduled task.

        • Fixed dates—Select a specific range of dates during which an orchestration job completed. When you select this option, you can use the calendar controls in the From and To fields to select specific dates.

      • Mode—The Mode options let you specify which types of orchestration jobs should be processed whenever the scheduled task runs.

        • Any—Select this option to process all jobs (batch and on-demand) that meet the criteria for the scheduled task.

        • Batch—Select this option to process only batch jobs that meet the criteria for the scheduled task.

        • On demand—Select this option to process only on-demand jobs that meet the criteria for the scheduled task.

      • Statuses—You can use the Statuses options to specify whether to process jobs that ran successfully or unsuccessfully as part of your scheduled task.

        • Complete—Select this option to process only orchestration jobs that completed successfully.

        • Error—Select this option to process only orchestration jobs that encountered errors when they ran.

        • Canceled—Select this option to process only orchestration jobs that have been canceled.

      Cancel non-finished jobs

      This task cancels jobs that are in a non-finished state. When you select this action, the following additional options become available:

      • Start—The Start options let you specify the start dates for the orchestration jobs that you want to process.

        • More than ... days before—Specify the number of days that an orchestration job started before the scheduled task.

        • More than ... hours before—Specify the number of hours that an orchestration job started before the scheduled task.

        • More than ... minutes before—Specify the number of minutes that an orchestration job started before the scheduled task.

        • Fixed dates—Select a specific range of dates during which an orchestration job started. When you select this option, you can use the calendar controls in the From and To fields to select specific dates.

      • Mode—The Mode options let you specify which types of orchestration jobs should be processed whenever the scheduled task runs.

        • Any—Select this option to process all jobs (batch and on-demand) that meet the criteria for the scheduled task.

        • Batch—Select this option to process only batch jobs that meet the criteria for the scheduled task.

        • On demand—Select this option to process only on-demand jobs that meet the criteria for the scheduled task.

      • Statuses—You can use the Statuses options to specify whether to process jobs that ran successfully or unsuccessfully as part of your scheduled task. Both options will be processed if you select neither check box or both check boxes.

        • Started—Select this option to wipe only the assets that have been started but not yet finished.

        • Waiting—Select this option to wipe only the assets that are in a waiting state.

        • Holding—Select this option to wipe only the assets that are in a holding state.

      Service call

      Initiates an orchestration job run at a regularly scheduled interval. This is done by sending an HTTP request to a published HTTPS input channel. The HTTPS input channel must use the input selector processor as an after-event in order to process the body of the HTTP request properly. The body of the HTTP request contains the input channel used to retrieve the driver file for triggering the job and the relative path to the driver file. Optionally, the retrieved driver file can be cached for a specified number of days. The input selector processor uses this information to make the driver file available to the orchestration job.

      When you select this action, the following additional options become available:

      • Service name—Select the HTTP connector to associate with the task.

      • Input channel—Select the input connector used to retrieve the file in the input path.

      • Input path—The path to the file to retrieve; this is the relative path to what is defined in the input channel.

      • Caching—Specify the number of days for the caching timeout.

      Note

      The Service name and Input channel lists are auto-populated with any published connectors in the flow context. For information about publishing flows, see section 9.2 "Publishing orchestration flows" in OpenText Communications - Web Client User Help (CCWC250100-H-UGD)section 9.2 "Publishing orchestration flows" in OpenText Communications - Web Client User Help (CCWC250100-H-UGD).

      To complete setting up a service call, follow these steps:

      1. Click Next.

      2. If you want to add one or more key-value pairs to the request header, click the Request headers button and fill out the following fields:

        • Key—Specify the key name for the request header element. This is the name of the orchestration system variable (created in Orchestrator).

          Note

          Keys in the header request are case-insensitive and must be unique. An error is issued if the user attempts to specify a duplicate key.

        • Value—Specify the value of the request header element. This is the value assigned to the orchestration system variable (created in Orchestrator).

      3. Click Add.

      4. Click OK.

      Sorting & bundling

      This task is used to initiate the post-sort run for the sorting and bundling element. That is, the orchestration service picks sorting assets from the repository that are part of the pre-sort run, merges files, and sorts the assets according to the settings of the sort element. A post-sort run then produces the final documents.

      When you select this action, the following additional options become available:

      • Created—Specify the creation dates for the orchestration jobs that you want to process.

        • More than ... days before—Specify the number of days that an orchestration job was created before the scheduled task.

        • More than ... hours before—Specify the number of hours that an orchestration job was created before the scheduled task.

        • More than ... minutes before—Specify the number of minutes that an orchestration job was created before the scheduled task.

        • Fixed dates—Select a specific range of dates during which an orchestration job was created. When you select this option, you can use the calendar controls in the From and To fields to select specific dates.

      • Dry run—A dry run is typically used to test settings. If enabled, the processing state of sorting assets is not affected after the dry run, and thus the assets can still be picked up for additional post-sort runs.

      • Consolidate multiple applications—This setting configures if batch aggregation or application consolidation is used during the post-sort run. If enabled, multiple communications can be selected and are consolidated in the post-sort run.

      • Flow model—Specify the ID of the flow model that produced the sorting assets in the pre-sort run.

      • Communication—Specify the ID of the communication that produced the sorting assets in the pre-sort run. If Consolidate multiple applications is checked, you can select multiple communications.

      • Pre-sort queue name—Specify the name of the pre-sort queue that produced the sorting assets in the pre-sort run. This can be used to pick up pre-sort assets for a certain desired output format.

      • Post-sort queue name—Specify the post-sort queue name to use in the post-sort run. If you don't specify a post-sort queue name, then all post-sort queues will be included in the run.

      Update expiration time

      When run, the Update expiration time task adjusts the earliest expiration time within all the waiting jobs by propagating upwards from flow steps, through flow instances, to the job. The expiration time can be seen by enabling the Expiration date column in the Monitor view. This option runs according to the cron expression that you specify and does not have additional settings associated with it.

      Wipe jobs

      This action removes from the database any qualifying jobs that have previously been marked for deletion. Additionally, wiping a job deletes any files and folders from NFS that are associated with the job. This option runs according to the cron expression that you specify and does not have additional settings associated with it.

      Wipe sorting assets

      This action removes any sorting assets from the database that have previously been marked for deletion.

      When you select this action, the following additional options become available:

      • Created—Specify the creation dates for the orchestration jobs that you want to process.

        • More than ... days before—Specify the number of days that an orchestration job was created before the scheduled task.

        • More than ... hours before—Specify the number of hours that an orchestration job was created before the scheduled task.

        • More than ... minutes before—Specify the number of minutes that an orchestration job was created before the scheduled task.

        • Fixed dates—Select a specific range of dates during which an orchestration job was created. When you select this option, you can use the calendar controls in the From and To fields to select specific dates.

      • Statuses—You can use the Statuses options to specify whether to process jobs that ran successfully or unsuccessfully as part of your scheduled task.

        • Processed—Select this option to wipe only the assets that have been processed.

        • Waiting—Select this option to wipe only the assets that are in a waiting state.

        • Expired—Select this option to wipe only the assets that are expired.

        • Canceled—Select this option to wipe only the assets that have been canceled.

      • Flow model—Specifies the ID of the flow model in the pre-sort run that produced the sorting assets that should be wiped.

      • Communication—Specify the ID of the communication in the pre-sort run that produced the sorting assets that should be wiped.

      • Pre-sort queue name—Specify the name of the pre-sort queue in the pre-sort run that produced the sorting assets that should be wiped.

    4. Click OK.

    5. Click Save.
